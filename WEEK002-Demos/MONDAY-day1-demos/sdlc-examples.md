# SDLC EXAMPLES FOR TEACHING
# Week 2 Monday - SDLC & Agile

This file contains examples and templates for teaching SDLC concepts.

---

## EXAMPLE 1: USER STORIES TEMPLATE

```
USER STORY FORMAT:
"As a [user type], I want [feature] so that [benefit]"

EXAMPLES - TO-DO APP:

✅ As a busy professional, I want to create tasks so that I don't forget important things

✅ As a student, I want to mark tasks complete so that I can track my progress

✅ As a parent, I want to set due dates so that I know when things are urgent

✅ As a team member, I want to share tasks with others so that we can collaborate

✅ As a forgetful person, I want reminders so that I don't miss deadlines
```

## EXAMPLE 2: MoSCoW PRIORITIZATION

```
MOSCOW METHOD - Prioritize Features:

M - MUST HAVE (Critical)
S - SHOULD HAVE (Important)
C - COULD HAVE (Nice to have)
W - WON'T HAVE (Not this sprint)

EXAMPLE - LIBRARY SYSTEM:

MUST HAVE:
- Add books to catalog
- Search books by title
- Check out books
- Return books

SHOULD HAVE:
- Search by author
- Track due dates
- Calculate late fees

COULD HAVE:
- Book recommendations
- Reading history
- Reserve books

WON'T HAVE (This Sprint):
- E-book integration
- Mobile app
- Email notifications
```

## EXAMPLE 3: SPRINT BACKLOG

```
SPRINT 01 - LIBRARY SYSTEM
Duration: 2 weeks
Sprint Goal: "Users can manage basic library operations"

COMMITTED TICKETS:
1. Add book to catalog (8 points)
2. Search books by title (5 points)
3. Check out book (5 points)
4. Return book (3 points)

Total: 21 story points
Team velocity: 20-25 points per sprint
Status: Realistic sprint scope ✅
```

## EXAMPLE 4: SDLC PHASES - INSTAGRAM REBUILD

```
PHASE 1: REQUIREMENTS
- Users can upload photos
- Users can follow others
- Feed shows photos from followed users
- Users can like and comment
- Photos have filters

PHASE 2: DESIGN
- Mobile-first (iOS & Android)
- Cloud storage (AWS S3)
- PostgreSQL database
- React Native framework
- Microservices architecture

PHASE 3: DEVELOPMENT
Team: 15 developers
- 5 Frontend (mobile apps)
- 5 Backend (APIs)
- 2 DevOps (infrastructure)
- 3 QA (testing)
Sprint-based (2-week sprints)

PHASE 4: TESTING
- Unit tests (individual functions)
- Integration tests (components together)
- UAT (real users test beta)
- Performance (handle 1M users?)

PHASE 5: DEPLOYMENT
- Canary deployment (5% users first)
- Monitor for crashes
- Roll out to 100% if stable

PHASE 6: MAINTENANCE
- Bug fixes (ongoing)
- New features (Stories, Reels)
- Security patches
- Performance optimization
```

## EXAMPLE 5: SPRINT PLANNING TEMPLATE

```
SPRINT PLANNING MEETING

1. REVIEW PRODUCT BACKLOG
   List all desired features

2. SELECT SPRINT SCOPE
   Choose realistic number of items

3. DEFINE SPRINT GOAL
   One sentence describing sprint outcome

4. BREAK DOWN TICKETS
   Split into subtasks with time estimates

5. COMMIT TO SPRINT
   Team agrees on deliverables

EXAMPLE:
Product Backlog: 20 features
Sprint Scope: Top 5 features
Sprint Goal: "Core task management functional"
Breakdown: Each feature → 3-5 subtasks
Commitment: Team confirms realistic
```

## EXAMPLE 6: DAILY STANDUP FORMAT

```
DAILY STANDUP (15 minutes max)

Each team member answers 3 questions:

1. What did I do yesterday?
   "Completed login feature, pushed to GitHub"

2. What will I do today?
   "Work on password reset functionality"

3. Any blockers?
   "Need help with email API integration"

RULES:
- Stand up (keeps it short!)
- No problem-solving (take offline)
- Same time every day
- Everyone participates
```

```

- Accepts completed work
```

## EXAMPLE 8: AGILE VS WATERFALL COMPARISON

```
WATERFALL:
✅ Clear structure
✅ Well-documented
✅ Fixed requirements
❌ Inflexible
❌ Late testing
❌ Users wait until end

AGILE:
✅ Flexible
✅ Continuous feedback
✅ Working software every sprint
✅ Early testing
❌ Less predictable
❌ Requires discipline

WHEN TO USE:
Waterfall → Government, construction software, regulated industries
Agile → Startups, tech companies, most modern software
```
